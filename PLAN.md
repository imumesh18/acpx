# PLAN

## Summary

Build `acpx` as one complete vertical slice: a thin ACP client wrapper, a
handwritten `agent_server` contract, a generated `agent_servers` snapshot from
the official ACP registry, a public `agent_servers` module that exposes raw
lookup and platform helpers over those servers, and an internal
single-shot `examples/cli.rs` harness.

## Status

- Completed: core runtime/error types plus fixture agent.
- Completed: subprocess-backed `Connection` and lifecycle tests.
- Completed: handwritten `AgentServer` contract plus manual command-backed
  server.
- Completed: registry sync generator plus merged `agent_servers.rs`.
- Completed: single-shot `examples/cli.rs`, example smoke coverage, and repo
  doc sync.

## Follow-up notes

- Revisit the transport boundary so consumers can provide their own
  `futures::AsyncRead` and `AsyncWrite` implementations directly, while keeping
  `async-process` isolated to subprocess launch helpers.
- Keep the binary-distribution installer story deferred until there is a clear,
  maintainable v1 design.

Do not fight the upstream ACP SDK's `!Send` model. The SDK is built around
local `!Send` futures and local task spawning, so the best implementation is to
make that explicit and contained. Keep the public API runtime agnostic, but
allow it to be `!Send`. Use Tokio only in the example CLI and tests.

## Key decisions

- Keep `futures` only where the ACP SDK forces it:
  - ACP I/O traits
  - local boxed task type expected by `ClientSideConnection`
- Prefer std async shapes in `acpx` where possible:
  - `std::future::Future`
  - local boxed `Task<'a, T>` alias
  - avoid `async-trait` in `acpx`
- Keep the public connection surface as `acpx::Connection`, which owns:
  - the inner `ClientSideConnection`
  - the spawned child process
  - ACP I/O task lifecycle
  - idempotent shutdown state
- `connect` only establishes transport and ACP wiring. It does not
  auto-initialize or auto-create sessions.
- `agent_server.rs` stays handwritten and trait-only.
- `agent_servers.rs` is the autogenerated public catalog module and includes
  raw lookup plus host-target helpers.

## Module and file plan

- `src/agent_server.rs`
  - Handwritten only
  - Contains `AgentServer`, `AgentServerMetadata`, runtime/local-task contract, and
    minimal shared launch types
  - No generated code
- `src/agent_servers.rs`
  - Fully autogenerated
  - Generated from the official ACP registry JSON
  - Contains concrete generated agent server definitions, lookup/listing APIs,
    and host-platform mapping over the committed registry snapshot
- `src/acpx.rs`
  - Handwritten
  - Contains `Connection`, subprocess wiring, session update capture, and
    lifecycle management
- `examples/cli.rs`
  - Handwritten
  - Internal single-shot integration harness

## Implementation plan

### Core runtime and connection

- Define a local task/runtime contract:
  - `Task<'a, T>` for dyn-safe async returns
  - `RuntimeContext` with explicit local task spawning
  - no Tokio types in the public library API
- Implement `acpx::Connection`:
  - wire child stdio into ACP
  - start ACP I/O
  - capture streamed `session/update` notifications
  - expose forwarded ACP agent methods
  - implement idempotent `close()` and safe drop cleanup
- Provide a built-in default ACP client handler for v0:
  - unsupported client-side ACP requests return `method_not_found`
  - session updates are captured and surfaced
  - richer client capability plumbing is deferred

### `agent_server` and generated servers

- Define the handwritten trait contract in `agent_server.rs`:
  - metadata access
  - `connect(runtime)` returning `acpx::Connection`
- Add one handwritten manual command-backed server first to validate the
  lifecycle before relying on generated servers.
- Add a generator exposed via `just registry-sync`.
- Implement the generator as a small Rust dev binary that:
  - fetches the official ACP registry JSON
  - normalizes launch data
  - rewrites `src/agent_servers.rs`
- Generated server rules:
  - `npx` and `uvx` entries become connectable generated servers
  - binary-only entries remain represented but return typed
    unsupported-launch errors in v0
  - no downloads, extraction, install cache, or runtime registry refresh

### `agent_servers.rs` catalog layer

- `agent_servers.rs` is the public generated catalog layer.
- Public catalog behaviors:
  - lookup by official registry id
  - iterate generated agents
  - resolve host platform for generated entries
  - expose typed unsupported-launch errors
- Do not add library-owned beautification or consumer-facing alias policy.
  Consumer applications can layer their own ids on top of raw registry ids.

### CLI contract

- The CLI shape is:
  - `cli.rs [agent-id-or-alias] [PROMPT] [options]`
- `agent-id-or-alias` resolution:
  - resolve via raw registry ids first
  - then example-local aliases such as `codex -> codex-acp`
- `PROMPT` is required in v0.
- The CLI is single-shot, not REPL-based.
- CLI options for v0:
  - `--cwd <path>`
  - `--mode <mode-id>`
  - `--permission-mode <mode-id>`
- CLI flow:
  - resolve agent
  - connect
  - initialize with fixed client info
  - create a new session with cwd
  - if `--mode` is set, call `session/set_mode`
  - if `--permission-mode` is set, map it to the agent's session config option
    API when available
  - send the positional prompt
  - print initialize result, auth methods, and streamed session updates
  - close cleanly
- Do not support generic config flags, load/resume flags, or model flags in
  v0.

### Delivery order

Land the full vertical slice in modular commits:

1. core task/runtime/error types plus fixture agent
2. `Connection` and lifecycle tests
3. handwritten `AgentServer` contract plus manual command-backed server
4. registry generator plus merged `agent_servers.rs`
5. single-shot `examples/cli.rs` and doc sync

## Test plan

- Unit test `Connection::close()` idempotence and drop behavior.
- Integration test against a tiny in-repo fixture ACP agent:
  - spawn
  - initialize
  - new session
  - optional mode/permission-mode setup
  - prompt
  - streamed update
  - shutdown
- Unit test handwritten manual server launch wiring.
- Unit test generator idempotence from a checked-in sample registry JSON
  fixture.
- Unit test `agent_servers.rs` lookup, iteration, and host-platform mapping.
- Unit test CLI-local alias behavior:
  - `codex` resolves to `codex-acp`
  - `droid` resolves to `factory-droid`
- Unit test binary-only generated servers return the correct
  unsupported-launch error.
- CLI smoke test via raw registry ids, CLI-local aliases, and the manual test
  agent path.
- Keep all tests offline and deterministic.

## Assumptions and defaults

- The best path is to embrace the upstream ACP SDK's local `!Send` execution
  model rather than hide it behind a custom runtime thread.
- Public API is runtime agnostic but not `Send`-guaranteed.
- `agent_server.rs` remains handwritten and trait-focused.
- `agent_servers.rs` is autogenerated and preserves the official ACP registry
  ids; any consumer-facing aliasing belongs outside the library.
- `--permission-mode` is a first-class CLI flag, but its implementation may
  internally use ACP session config APIs depending on what the target agent
  exposes.
- Pre-`1.0.0` API changes remain acceptable if implementation proves a better
  contract.
